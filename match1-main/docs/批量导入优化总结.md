# 批量导入优化总结

## 问题背景

原系统在批量导入大量数据时遇到Prisma事务超时错误：
- 错误代码：`P2028`
- 默认超时：5000ms (5秒)
- 实际执行时间：超过5276ms
- 根本原因：事务内循环操作导致数据库往返次数过多

## 优化方案

采用**方案2（优化批量操作）+ 方案4（全局配置）**的组合解决方案：

### 1. 全局Prisma配置优化

**文件**: `src/lib/prisma.ts`

```typescript
export const prisma = new PrismaClient({
  log: process.env.NODE_ENV === 'development' ? ['query', 'error', 'warn'] : ['error'],
  // 配置全局事务选项
  transactionOptions: {
    maxWait: 10000, // 最大等待时间 10秒
    timeout: 30000, // 事务超时时间 30秒
  },
});
```

### 2. 参与者批量导入优化

**文件**: `src/app/api/participants/batch-import/route.ts`

#### 优化前的问题
- 在事务中使用 `for` 循环逐个处理
- 每条记录执行3次数据库操作：检查重复 → 创建记录 → 创建审计日志
- 数据库操作次数：`N × 3`（N为记录数）

#### 优化后的解决方案
1. **预处理数据验证** - 事务外进行基本验证
2. **批量重复检查** - 一次查询检查所有重复数据
3. **批量创建操作** - 使用 `createMany` 一次性创建
4. **批量审计日志** - 使用 `createMany` 一次性创建
5. **增加事务超时** - 20秒特定超时

#### 性能提升
- 数据库操作次数：`4`（固定，不随记录数增长）
- 事务时间：大幅减少
- 超时风险：从高风险降至低风险

### 3. 节目批量导入优化

**文件**: `src/app/api/programs/batch-import/route.ts`

#### 优化前的问题
- 更复杂的循环操作：验证 → 检查重复 → 查找选手 → 创建选手 → 创建节目 → 更新关联 → 创建审计日志
- 每个节目可能触发多次选手查找和创建
- 数据库操作次数：`N × (3 + M × 2)`（N为节目数，M为平均选手数）

#### 优化后的解决方案
1. **预处理数据** - 验证并解析选手姓名
2. **批量检查重复节目** - 一次查询检查所有重复
3. **批量查找选手** - 一次查询获取所有现有选手
4. **批量创建新选手** - 使用 `createMany` 创建缺失选手
5. **批量创建节目** - 使用 `createMany` 创建所有节目
6. **分批更新关联** - 50个一批更新选手关联
7. **批量审计日志** - 一次性创建所有日志
8. **增加事务超时** - 60秒（节目导入更复杂）

#### 性能提升
- 数据库操作次数：`7 + ⌈关联数/50⌉`（大幅减少）
- 事务时间：显著减少
- 支持更大批量导入
